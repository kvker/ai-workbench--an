# `/reports` 使用看板方案

## 1. 目标

`/reports` 页面用于回答一个问题：

> AI 工作台现在有多少人在参与使用，分别以什么角色参与，需求推进到什么阶段。

第一版不做复杂埋点，不统计登录活跃、页面访问、留存率等行为数据。先基于现有需求数据，做一版低开发成本、可解释、可落地的使用看板。

## 2. 做了什么

新增一个“使用看板”页面，包含三个部分：

| 区域 | 内容 |
|------|------|
| 顶部指标 | 总流程需求数、创建用户数、负责用户数、干系用户数 |
| 图表一 | 每个用户的创建数、负责数、关联数 |
| 图表二 | 每个用户创建的需求当前分布在哪个阶段 |
| 明细表 | 按用户查看参与数量和阶段分布 |

页面会展示全部启用用户。用户较多时，图表在容器内左右滚动，不影响整个页面布局。

## 3. 能看什么

### 3.1 谁在创建需求

通过“创建数”可以看到哪些用户在主动把需求录入 AI 工作台。

用途：

- 判断工作台是否被真实用于需求入口。
- 看出主要需求输入人是谁。
- 发现是否只有少数人在创建需求。

### 3.2 谁在负责需求

通过“负责数”可以看到哪些用户在承接需求。

用途：

- 判断需求责任是否集中在少数人身上。
- 辅助观察团队承接负载。
- 识别有人创建但无人负责、或负责人分布异常的情况。

### 3.3 谁被关联进需求协作

通过“关联数”可以看到哪些用户作为干系人参与需求。

用途：

- 判断 AI 工作台覆盖到了哪些协作人员。
- 看需求是否形成多人协作，而不是只有创建人和负责人。
- 辅助观察跨角色协作范围。

### 3.4 需求推进到了哪里

通过“创建需求阶段分布”可以看到每个创建人名下的需求当前处于：

- 产品
- 前端
- 后端
- 测试
- 归档

用途：

- 判断需求是否卡在某个阶段。
- 看不同创建人的需求推进情况。
- 辅助发现“创建很多但推进不足”的情况。

## 4. 有什么用

这版看板主要服务三个判断：

| 判断 | 看板如何支持 |
|------|--------------|
| 有多少人在参与 AI 工作台 | 看创建用户、负责用户、干系用户数量 |
| 用户以什么方式参与 | 看创建数、负责数、关联数 |
| 需求是否真正向后推进 | 看五阶段分布 |

它不是传统 BI，也不是用户行为分析平台。它更像一个轻量的“需求协作使用情况”入口，用来快速判断 AI 工作台是否已经进入团队的实际流程。

## 5. 口径说明

当前统计的是：

- 未删除需求。
- 已进入 AI Native 流程的需求。
- 启用状态的用户。

三个核心指标含义：

| 指标 | 含义 |
|------|------|
| 创建数 | 用户作为创建人创建的需求数量 |
| 负责数 | 用户作为负责人承接的需求数量 |
| 关联数 | 用户作为干系人被关联的需求数量 |

阶段分布的含义：

> 按创建人统计其创建需求当前推进到哪个阶段。

注意：阶段分布不代表创建人本人亲自推进了这些状态，只代表他创建的需求当前状态。

## 6. 当前边界

这版看板能反映“需求协作使用情况”，但不能完整代表“真实用户活跃度”。

暂时不能回答：

- 今天有多少人登录。
- 多少人打开过页面。
- 多少人使用过 AI 会话。
- 谁亲自推进了流程状态。
- 用户留存和使用频率。

这些需要后续补充行为埋点或状态变更历史。

## 7. 后续方向

如果后续要进一步判断“真实活跃使用”，建议补充：

| 方向 | 价值 |
|------|------|
| 登录和页面访问记录 | 统计真实活跃用户 |
| AI 会话使用记录 | 判断 AI 能力是否被使用 |
| 流程状态变更历史 | 统计谁在推进流程 |
| 后端聚合接口 | 提升报表性能和权限控制 |

## 8. 结论

第一版 `/reports` 使用看板的价值是：

> 用现有需求数据，低成本看清 AI 工作台在团队里的协作覆盖情况。

它可以先回答“哪些人在参与、以什么角色参与、需求推进到了哪里”，为后续是否补充用户行为埋点提供依据。
