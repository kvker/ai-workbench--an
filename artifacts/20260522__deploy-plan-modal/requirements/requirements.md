# 发布计划弹框需求文档

## 背景与目标

当前详情页需要把“发布计划”作为信息区能力暴露出来，用户可以查看当前需求关联的发布计划，并为尚未创建发布计划的工程添加发布计划。

本次任务还要求调整现有自动分支生成方案：保留现有分支名生成方法，但不再由该方法直接关联或创建分支；真正的分支创建需要通过 devops 的“创建发布计划”接口完成。

## 验收标准 (AC)

- [x] AC1: 详情页信息区展示“发布计划”按钮。
- [x] AC2: 点击“发布计划”后出现弹框，弹框内竖向展示当前需求关联的发布计划列表。
- [x] AC3: 发布计划列表卡片横向展示工程名和分支名。
- [x] AC4: 获取详情页数据时，同步获取当前需求相关发布计划列表信息。
- [x] AC5: 发布计划列表查询使用 frontend 工程相关的发布计划查询能力。
- [x] AC6: 弹框底部提供加号入口，用于添加新的发布计划。
- [x] AC7: 点击加号后显示工程选择框，只允许选择当前需求尚未创建发布计划的工程。
- [x] AC8: 新增发布计划时不传 `branchName`，由 `POST /deployPlan/add` 负责生成和创建分支；现有自动分支名生成方法仅作为旧工作区兜底保留。
- [x] AC9: 调用创建发布计划接口时至少提交 `issueId`、`projectConfigId`，其余字段按现有接口能力和页面上下文处理。
- [x] AC10: 创建发布计划成功后刷新弹框内发布计划列表，用户能看到新增工程和分支。
- [x] AC11: 发布计划列表获取成功后，根据每个列表项对应的仓库和分支，在本地目录准备代码仓库。
- [x] AC12: 若本地已存在对应仓库，则进入该仓库执行 `git pull --rebase` 更新目标分支。
- [x] AC13: 若本地不存在对应仓库，则按发布计划列表项中的仓库信息和分支执行 `git clone`。
- [x] AC14: 发布计划卡片右侧提供解除按钮，点击后二次确认，确认后解除当前发布计划绑定并刷新列表。

## 范围

### 包含

- 详情页信息区新增发布计划入口。
- 发布计划弹框、列表展示、添加入口和工程选择交互。
- 详情页加载流程增加发布计划列表获取。
- 前端 service 层补充发布计划列表查询和创建发布计划调用。
- 调整现有自动分支生成调用链，使其只生成分支名，不直接关联分支。
- service 侧在发布计划列表获取成功后同步准备 `repos/` 下对应仓库分支。

### 不包含

- 发布计划详情页、进度详情页或审批/测试状态完整管理。
- 发布计划删除、修改、审核、测试等后续操作。
- 后端 devops-service 接口实现改造。
- 登录、权限、用户身份体系调整。

## 依赖

- devops “创建发布计划”接口：`POST /deployPlan/add`。
- devops 发布计划列表查询接口：优先使用当前 frontend 工程需要的发布计划查询能力，具体路径在设计阶段结合现有前端代码确认。
- 详情页当前需求数据中需要能拿到 `issueId`。
- 页面需要能获取可选工程列表，具体来源在设计阶段结合现有前端代码确认。

## 涉及模块

- 前端详情页信息区。
- 前端发布计划相关 service/API 封装。
- 当前需求详情数据加载流程。
- 自动分支名生成与创建发布计划衔接逻辑。
- service 本地仓库 clone / pull --rebase 能力。

## 开放问题

- “获取发布计划的列表查询 frontend 的”需要在代码中确认 frontend 工程的 projectConfigId 来源与筛选方式。
- 创建发布计划时 `planDeployDate`、`reviewUser`、`testUser`、`remark` 是否必填，需要以现有接口和运行行为确认；若页面缺少来源，默认不在本次交互中新增额外表单字段。
- 工程选择框的数据源需要在代码中确认：是否已有项目配置列表接口或详情页已有项目配置数据。
- 发布计划列表项中仓库 URL、仓库名、分支名字段需要在代码或接口返回中确认。
- `repos/` 下本地仓库目录命名规则需要在设计阶段确认，避免与当前 `repos/app`、`repos/service` 工程冲突。
- 若本地仓库已有未提交变更，是否允许执行 `git pull --rebase` 需要在设计阶段制定保护策略。
