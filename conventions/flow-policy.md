# 流程轻重策略

AI 应根据风险和不确定性选择最轻但足够安全的流程。目标是减少不必要确认，同时避免高风险改动失控。

## 流程等级

| 等级 | 名称 | 适用场景 | 最低产出 | 是否先确认 |
|------|------|----------|----------|------------|
| L0 | 直接修复 | 拼写、注释、明显一行 bug、无行为风险的小配置 | 最终总结 + 验证结果 | 否 |
| L1 | 快速修复 | 单模块、小范围、行为明确、有现成验证 | 原始输入 + 实现 + 测试 | 否，除非发现风险 |
| L2 | 标准开发 | 新增小功能、明确方案、影响少量模块 | 原始输入 + 技术规范 + 实现 + 测试 + 质量评价 | 技术规范后确认，可在用户要求加速时跳过 |
| L3 | 完整流程 | 需求不清、跨模块、架构、接口、数据模型、权限、支付、数据迁移 | 全阶段 + 质量评价 | 每个关键阶段确认 |

## 升级条件

满足任一条件时，至少使用 L2；高风险时使用 L3：

- 修改公开接口、数据库结构、认证鉴权、支付、权限、日志审计、部署配置。
- 影响两个以上工程或多个核心模块。
- 需求存在开放问题或验收标准不明确。
- 需要新增依赖、迁移数据、运行破坏性命令或访问外部服务。
- 测试缺失且改动会影响用户可见行为。

## 降级条件

满足全部条件时，可以使用 L0 或 L1：

- 用户目标明确，无开放问题。
- 改动局部且可回滚。
- 不涉及数据、安全、权限、外部接口或部署。
- 有可运行的最小验证，或能明确说明无法运行的原因。

## 确认策略

- L0/L1：可以先做，完成后报告改动和验证。
- L2：默认在技术规范后确认；若用户明确说“直接改”“快速处理”，可以先实现，但必须记录跳过确认的原因。
- L3：不得静默跳过需求和方案确认，除非用户明确授权并接受风险。
- 任何等级遇到新增高风险事实，立即升级流程并说明原因。

## 文档策略

- L0 不强制创建 artifacts。
- L1 可以创建轻量 artifacts；若任务很小，也可只在最终总结中记录。
- L2/L3 必须创建 artifacts 产出目录并更新根 `AGENTS.md` 活跃 Artifacts 表。
- 完成 L2/L3 后应运行 `/an-eval`，将报告保存到 `testing/eval-report.md`。
