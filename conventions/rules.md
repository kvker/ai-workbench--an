# 对话长期记忆规则

## 目标

AI 在任意对话过程中持续感知用户是否正在“纠错、约束、提升”AI 的工作方式。若满足长期沉淀条件，应写入 `conventions/memories/` 中对应功能主题文件；若同名主题已存在，则追加到该文件。

该机制属于对话层常驻行为，不属于 `raw-input → requirements → design → tech-spec → implementation → testing → deployment` 开发流程的一部分，也不要求为每次沉淀创建 artifact。

## 触发时机

| 时机 | 行为 |
|------|------|
| 用户纠正 AI 的理解、行为或产出 | 立即判断是否属于“纠错” |
| 用户提出“以后都要 / 不要 / 必须 / 禁止”等表达 | 立即判断是否属于“约束” |
| 用户指出更好的协作方式、流程或质量标准 | 立即判断是否属于“提升” |
| 一件事情自然结束时 | 快速回看本轮是否有未处理的长期记忆信号 |

## 分类

| 分类 | 含义 | 示例信号 |
|------|------|----------|
| 纠错 | 修正 AI 已发生的错误、误判或遗漏 | “你理解错了”“不是这个意思”“以后别再...” |
| 约束 | 增加未来行为边界、禁止项或必须项 | “所有对话都要...”“不要...”“必须先...” |
| 提升 | 形成更高质量、更高效率的稳定做法 | “以后可以自动...”“这个判断应该常规化” |

## 写入条件

同时满足以下条件才可沉淀为长期记忆：

- 来自用户明确表达，或来自用户确认过的对话事实。
- 对未来多轮对话或多个任务有复用价值，不只是当前一次性选择。
- 表述为可执行行为，而不是模糊偏好。
- 不与现有规则冲突；如冲突，先向用户报告。
- 不包含密钥、凭据、隐私数据、临时路径或不可泛化的上下文。

## 不写入条件

任一条件满足时，不写入长期规则：

- 只是本次任务的业务需求或临时方案。
- 用户表达不明确，无法判断是否希望长期生效。
- 只适用于单次调试现场或单个 artifact。
- 与现有规则冲突且用户尚未确认取舍。
- 会导致 AI 过度自动化、跳过必要确认或扩大权限。

## 对话行为

AI 在对话中发现候选长期记忆时，应按以下顺序处理：

- 先判断分类：纠错、约束、提升或不沉淀。
- 能明确满足写入条件时，在 `conventions/memories/` 中找到对应功能主题文件并追加；不存在时创建功能性命名文件。
- 判断不充分时，先向用户确认，不要擅自长期化。
- 明确不满足写入条件时，不需要打断当前对话，也不需要写入记录。
- 若用户正在推进其他任务，沉淀动作应尽量小而快，避免喧宾夺主。

## 写入规则

- `conventions/rules.md` 只保存长期记忆机制的指导规则，不作为具体记忆内容的回写目标。
- `conventions/memories/` 保存具体优化记录、来源、分类、判断依据和后续追加内容。
- memory 文件使用功能性名字，不带日期，例如 `dialog-memory.md`、`git-workflow.md`。
- 写入前先查找同名或同主题 memory 文件；找到则追加，找不到才创建新文件。
- 不要把对话流水直接追加到 `rules.md`；具体记录写入 `memories/`。
- 只写行为规则和必要来源，不写用户隐私、临时上下文或一次性任务细节。
- 从代码反推的信息不得直接作为对话长期记忆；必须先获得用户确认。

## 自动读取

根 `AGENTS.md` 只存路由。AI 开始任意对话或任务时应按需读取本规则；在对话过程中持续应用，而不是等到开发流程结束才执行。
