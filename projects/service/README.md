# AI Workbench Service

这是 AI Native 工作台的服务端入口。

工作台的前端负责呈现需求页、详情页、流程区、对话区和代码页；本服务负责承接浏览器不适合直接完成的能力，例如会话编排、工程文件夹操作、命令执行、在线编辑器启动、需求元信息读写，以及后续与 Codex 能力的集成。

## 定位

本服务不是业务页面，也不是 workspace-template 本身。

它是工作台和服务器资源之间的薄服务层：

- 为每个需求定位对应的工程文件夹、需求分支和用户工作区。
- 为详情页提供需求元信息、流程状态、文档入口和代码页入口。
- 为对话区承接 AI 会话、工具调用、文件读写和执行过程展示。
- 为代码页管理 vscode-server 等在线工程能力的启动、鉴权和访问地址。
- 将 AI、用户和流程产生的结果沉淀回当前需求的工程文件夹。

## 架构背景

工作台采用“一个模板仓库 + 多个需求分支 + 多个用户工作区”的模型。

一个需求对应一个需求分支。多人协作同一需求时，可以在服务器上拥有各自的实体工作区；这些工作区检出同一个需求分支，但通过文件夹隔离本地配置、缓存、编辑器状态和临时产物。

服务端的核心边界是当前用户、当前需求、当前工作区。任何文件、命令、会话和代码页操作都应落在这个边界内。

## 当前状态

当前代码是一个最小 Express 服务：

```bash
npm run dev
```

默认监听 `3100` 端口，并提供 `/` 健康检查式响应。原型阶段的工作台数据保存在 `data/workbench.json`，前端通过 `/api/workspace` 和 `/api/task` 读取。

## 约定

- 服务端接口优先围绕详情页能力设计，而不是围绕单个 UI 组件堆接口。
- API 返回结构应清楚表达需求、流程、会话、文件和执行状态之间的关系。
- 涉及文件系统和命令执行的能力必须显式绑定工作区，避免跨需求或跨用户操作。
- 用户私有配置、临时文件、缓存和编辑器状态不应写入共享需求分支。
- 具体实现可以逐步演进，但服务层应保持薄、明确、可审计。
