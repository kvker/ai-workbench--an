## 项目定位

当前目录是 AI Native 工作台的服务端。它负责在 Web 前端能力之外，承接需要服务器权限或长期状态的工作：

- 需求详情页的工程文件夹和元信息管理。
- 需求分支、用户工作区和文件系统边界管理。
- 对话区所需的 AI 会话、工具调用、文件读写和执行过程编排。
- 代码页所需的 vscode-server 启动、鉴权、访问地址和工作目录绑定。
- 流程节点状态、节点文档入口和交付产物的持久化。

不要把它写成前端页面项目，也不要把它写成 workspace-template 的业务实现。这里应该保持为薄服务层。

关键概念：

- 需求页：只负责需求列表和入口。
- 详情页：单个需求的执行空间，本质上绑定一个带元信息的工程文件夹。
- 流程区：展示需求分析、原型编写、文档移交、交付等节点及状态。
- 对话区：在线版 AI 对话 IM，优先复用 Codex Web / Codex SDK 能力。
- 代码页：当前需求绑定的 vscode-server 在线编辑器。
- 需求分支：一个分支代表一个需求。
- 用户工作区：服务器上的实体文件夹，用于隔离同一需求下不同用户的本地状态。

## 工程原则

- 所有能力都必须显式绑定当前需求和当前用户工作区。
- 文件读写、命令执行、编辑器启动等高风险能力必须限制在当前工作区内。
- 不要把用户私有配置、skills、缓存、编辑器状态和临时产物当作共享需求上下文。
- 服务端接口应表达业务边界，而不是泄漏过多页面细节。
- 优先实现清晰的小接口和可审计的数据结构，再扩展复杂编排。
- 保持 CommonJS / Node / Express 的现有风格，除非项目明确迁移。
- 不要引入大型框架或复杂抽象来解决当前还不存在的问题。

## 建议目录演进

当前只有 `app.js`。后续可按能力自然拆分：

```text
app.js
src/
  routes/
  services/
  repositories/
  workspace/
  codex/
  vscode/
```

其中：

- `routes/` 只处理 HTTP 入参、出参和错误转换。
- `services/` 编排需求、流程、会话和代码页能力。
- `repositories/` 读写需求元信息和流程状态。
- `workspace/` 处理工作区定位、路径校验、文件访问和命令执行边界。
- `codex/` 封装 Codex Web / SDK 对接。
- `vscode/` 封装 vscode-server 生命周期和访问地址。

不要为了目录完整性提前创建空模块。只有出现真实职责时再拆。

## API 设计倾向

优先围绕以下资源建模：

- `demands`：需求项和详情页入口。
- `workspaces`：当前需求下的用户工作区。
- `flows`：流程节点、状态和文档入口。
- `sessions`：AI 对话会话和历史。
- `code-page`：vscode-server 入口、状态和工作目录。

返回数据应包含足够的状态字段，方便前端展示“进行中、待确认、已完成、需修改”等流程状态。

## 验证

修改服务端后至少执行：

```bash
node app.js
```

如果增加了 npm scripts，优先使用项目脚本验证。若新增文件系统、命令执行或在线编辑器相关能力，需要额外验证路径边界和错误分支。
